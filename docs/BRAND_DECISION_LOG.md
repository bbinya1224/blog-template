# 브랜드 의사결정 기록

> 이 문서는 BlogToneLab에서 오롯이(Orot-i)로의 리브랜딩 과정에서 내린 주요 결정들과 그 이유를 기록합니다.

---

## 1. 왜 리브랜딩이 필요했는가

### 기존 문제점

**BlogToneLab**이라는 이름은 다음과 같은 한계가 있었습니다.

| 문제                 | 설명                                                               |
| -------------------- | ------------------------------------------------------------------ |
| **기능 중심 네이밍** | "블로그 톤 + 랩"은 기술 도구처럼 들림. 사용자가 아닌 기능이 주인공 |
| **타겟 제한**        | "Blog"가 이름에 들어가면서 블로거만을 위한 도구로 인식됨           |
| **용도 제한**        | "리뷰 생성기"로 포지셔닝되어 맛집 리뷰에 한정된 느낌               |
| **감성 부재**        | 영문 기술 용어 조합은 차갑고 사무적인 인상                         |

### 핵심 인식의 전환

앱을 만들고 사용하면서 깨달은 것은, 이 도구의 본질이 "리뷰 생성"이 아니라 **"경험 증폭"**이라는 점이었습니다.

```
이전 인식: 블로그 톤을 분석하고 → 리뷰를 자동 생성하는 도구
새로운 인식: 대화를 통해 세부 기억을 끌어내고 → 경험을 풍부하게 기록하는 도구
```

사용자가 채팅을 통해 자신의 경험을 이야기할 때, AI의 질문이 잊혀진 디테일을 되살리는 과정 자체가 이 도구의 핵심 가치였습니다. "자동 생성"이 아니라 "경험 증폭"이 본질입니다.

---

## 2. 네이밍 결정 과정

### 방향 설정

새 이름은 다음 조건을 만족해야 했습니다:

1. **사용자가 주인공**이어야 한다 (기능이 아니라)
2. **감성적**이어야 한다 (기술 도구가 아닌 기록 동반자)
3. **한글** 기반이어야 한다 (주 사용자가 한국어 사용자)
4. **확장 가능**해야 한다 (맛집에 한정되지 않는)
5. **발음이 쉽고 기억에 남아야** 한다

### 최종 선택: 오롯이

**"오롯이"** — 순우리말 부사 "오롯이(온전히, 빠짐없이)"에서 가져옴.

선택 이유:

- "온전히 내 경험을 담다"라는 앱의 철학을 한 단어로 표현
- 한글 고유의 부드러움이 감성적 톤에 부합
- 영문 표기 "Orot-i"가 자연스러움 (도메인 확보 가능)
- "-이" 접미사가 캐릭터/동반자 느낌을 줌 (친근함)
- 맛집/제품/독서 등 어떤 카테고리에도 어울림

### 슬로건

```
"경험은 당신이, 표현은 오롯이가"
```

- 주어를 나누어 역할 분담을 명확히 함
- 사용자가 주인공, 오롯이는 조력자
- 대구법으로 리듬감 있고 기억에 남음

## 3. 포지셔닝 전환

| 항목        | 이전 (BlogToneLab) | 이후 (오롯이)             |
| ----------- | ------------------ | ------------------------- |
| 정체성      | 리뷰 생성기        | 경험 증폭 기록 도구       |
| 타겟        | 블로거             | 블로거 + 일반 기록자      |
| 범위        | 맛집 리뷰          | 맛집 / 제품 / 뷰티 / 독서 |
| 톤          | 기술적, 사무적     | 감성적, 문학적            |
| 사용자 역할 | 입력자             | 경험의 주인공             |
| AI 역할     | 생성기             | 기억을 끌어내는 동반자    |

### 카테고리 확장 결정

기존에는 "맛집" 하나만 활성화되어 있었습니다. 리브랜딩과 함께 카테고리 구조를 재설계했습니다.

| 카테고리 | 결정                | 이유                                             |
| -------- | ------------------- | ------------------------------------------------ |
| 맛집     | 유지 (활성)         | 기존 핵심 기능, 이미 검증됨                      |
| 제품     | 추가 (준비중)       | 맛집 다음으로 리뷰 수요가 높은 카테고리          |
| 뷰티     | 유지 (준비중)       | 기존에도 계획되어 있던 카테고리                  |
| 독서     | 라벨 변경 "책→독서" | "책"보다 "독서"가 경험 기록이라는 맥락에 더 부합 |

## 4. 디자인 시스템 결정

### 컬러 팔레트 선택 근거

기존 ChatContainer 컴포넌트가 이미 `stone` + `orange` 계열을 사용하고 있었습니다. 이것이 앱에서 가장 잘 작동하는 부분이었기 때문에, 이 팔레트를 앱 전체로 확장하기로 결정했습니다.

```
이전: 앱 전체는 blue/slate → ChatContainer만 stone/orange
이후: 앱 전체를 stone/orange 기반 Warm & Soft로 통일
```

| 컬러                  | 값         | 선택 이유                                  |
| --------------------- | ---------- | ------------------------------------------ |
| Warm Coral (#E8825C)  | Primary    | 따뜻함 + 활력. 차가운 blue-500 대비 감성적 |
| Soft Amber (#F4A261)  | Secondary  | Coral과 자연스러운 조화. 호버/강조용       |
| Warm White (#FFFBF7)  | Background | 순백(#fff)보다 따뜻하고 편안한 느낌        |
| Peach Cream (#FFF5ED) | Surface    | 카드 배경에 미세한 따뜻함 부여             |
| Stone 700 (#44403C)   | Text       | 순흑(#000)보다 부드러운 가독성             |

### 왜 blue를 버렸는가

Blue는 신뢰/전문성을 전달하지만, "경험 기록 도구"에는 맞지 않았습니다.

- Blue: 차갑다, 사무적이다, SaaS/B2B 느낌
- Warm Coral: 따뜻하다, 개인적이다, 일기장/노트 느낌

사용자가 자신의 경험을 편안하게 꺼내놓을 수 있으려면, 도구가 아니라 **대화 상대** 같은 느낌이어야 했습니다.

### 타이포그래피

| 결정                       | 이유                                                         |
| -------------------------- | ------------------------------------------------------------ |
| Pretendard 우선 (fallback) | 현대적이면서 부드러운 한글 서체. 감성 톤에 부합              |
| Inter 유지                 | 영문에서 가장 가독성 좋은 sans-serif. 변경 불필요            |
| Noto Sans KR 유지          | Google Fonts에서 안정적으로 로드됨. Pretendard의 안전망 역할 |

## 5. 기술적 구현 결정

### CSS 변수 기반 접근

색상을 Tailwind 하드코딩이 아닌 CSS 변수(`var(--primary)` 등)로 관리하기로 결정했습니다.

이유:

- 향후 다크모드 대응 시 변수만 바꾸면 됨
- 브랜드 컬러 변경 시 한 곳만 수정
- globals.css가 디자인 토큰의 단일 진실 공급원(Single Source of Truth) 역할

### 변경하지 않은 것들

의도적으로 변경하지 않은 영역들:

| 영역                       | 이유                                              |
| -------------------------- | ------------------------------------------------- |
| 프롬프트 시스템 (Supabase) | DB 데이터는 별도 마이그레이션 필요. 브랜딩과 분리 |
| API 라우트                 | 내부 로직은 브랜드와 무관                         |
| 인증 로직                  | NextAuth 설정은 기능적 영역                       |
| 카테고리 config 확장       | 프롬프트/로직 변경이 필요한 별도 작업             |

이유: 리브랜딩은 "사용자에게 보이는 것"에 집중해야 합니다. 내부 로직 변경과 섞으면 롤백이 어렵고, 버그 추적이 복잡해집니다.

## 6. Welcome 화면 텍스트 결정

### 인사말 변경

```
이전: "안녕하세요" / "어떤 기록을 남길까요?"
이후: "{사용자명}님, 안녕하세요" / "오늘은 어떤 경험을 남겨볼까요?"
```

변경 이유:

- **개인화**: 이름을 불러주는 것만으로 친밀감 상승
- **"기록" → "경험"**: 행위(기록)가 아니라 소재(경험)에 초점
- **"오늘은"**: 시간적 맥락을 더해 일상적 사용 유도
- **"남겨볼까요?"**: "남길까요"보다 부드럽고 권유하는 톤

### 로그인 유도 메시지 변경

```
이전: "어서오세요! 제가 사용자님에게 좀 더 도움을 주기 위해 로그인이 필요해요."
이후: "반가워요! 오롯이와 함께 경험을 기록하려면 로그인이 필요해요."
```

변경 이유:

- 브랜드명 노출로 인지도 형성
- "도움을 주기 위해" → "함께 경험을 기록하려면"으로 동반자 톤 강화
